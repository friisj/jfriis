# Luv Stage — Scene Pipeline Spec

> Concrete specification for building chassis module scenes, starting with the eye module.

## Design Decisions

These decisions were made through alignment discussion and inform everything below.

1. **One scene per pipeline step** — each step is its own scene component
2. **Artifacts extend module media** — `artifact_type`, `source_scene_slug`, `generation_metadata` added to `luv_chassis_module_media` (schema deferred until artifacts are produced)
3. **Providers**: Hugging Face, Replicate, Google models (Imagen, etc.)
4. **Manual/free navigation** — no enforced pipeline order; progress is implicit from existing artifacts
5. **Scene contracts DB-declared** — `input_schema`/`output_schema` JSON columns on `luv_scenes`
6. **Extensible toolbar slots** — scenes export React components injected into StagePlayer toolbar
7. **Agent integration: read-only first** — agent observes scene state via `setPageData()`; bidirectional API designed but not implemented
8. **Prompts sourced from module params** — chassis parameters auto-compose into a starting prompt; user refines
9. **First scene to build: prompt playground** — full workbench (template editor + generation + annotation + prompt versioning)

## Scene Contract

### DB Schema Extension (`luv_scenes`)

Add two JSONB columns:

```sql
alter table luv_scenes
  add column input_schema jsonb default '[]'::jsonb,
  add column output_schema jsonb default '[]'::jsonb;

comment on column luv_scenes.input_schema is
  'Array of artifact type descriptors this scene consumes. Empty = no inputs required.';
comment on column luv_scenes.output_schema is
  'Array of artifact type descriptors this scene produces.';
```

Schema shape:

```ts
interface ArtifactTypeDescriptor {
  type: string;           // e.g. 'image', 'video', '3d-model', 'prompt-template'
  artifact_type?: string; // e.g. 'canonical', 'candidate', 'reference'
  description: string;
  required: boolean;
}
```

This enables querying "which scenes can run given what artifacts exist" without loading components.

### Component Contract

Every scene component receives `SceneProps` (already defined in `lib/luv/stage/types.ts`):

```ts
interface SceneProps {
  descriptor: SceneDescriptor;
  chassisModules: ModuleContext[];
  templateContext: TemplateContext;
}
```

Scenes may additionally export a toolbar extension:

```ts
// Optional: toolbar controls injected into StagePlayer
export function SceneToolbar({ descriptor, state }: SceneToolbarProps): ReactNode;
```

### Toolbar Slot API

StagePlayer provides three named slots in its toolbar: `left`, `center`, `right`.

```ts
interface SceneToolbarProps {
  descriptor: SceneDescriptor;
  state: unknown;  // scene-specific state, opaque to player
}
```

The scene component exports a `SceneToolbar` named export. StagePlayer checks for it via the component registry and renders it alongside the base toolbar (scene name, category badge, scene switcher, unload).

Implementation: extend `component-registry.ts` to track toolbar components:

```ts
interface SceneRegistryEntry {
  component: ComponentType<SceneProps>;
  toolbar?: ComponentType<SceneToolbarProps>;
}
```

## Artifact Model

### Phase 1: No schema changes

The prompt playground produces prompt templates — these are stored as scene-local state (React state + optionally persisted to a JSONB field or new table when versioning is needed).

### Phase 2: When generation scenes produce real media

Add columns to `luv_chassis_module_media`:

```sql
alter table luv_chassis_module_media
  add column artifact_type text default 'reference',
  add column source_scene_slug text,
  add column generation_metadata jsonb;

comment on column luv_chassis_module_media.artifact_type is
  'reference | candidate | canonical — lifecycle stage of this media item';
comment on column luv_chassis_module_media.source_scene_slug is
  'Scene that produced this artifact (null for manually uploaded media)';
comment on column luv_chassis_module_media.generation_metadata is
  'Provider, model, prompt, parameters, timing, etc.';
```

**artifact_type values:**
- `reference` — manually uploaded reference images (current behavior, default)
- `candidate` — generated by a generation scene, not yet evaluated
- `canonical` — promoted through reinforcement, approved for downstream use

## Eye Module Pipeline

All scenes below target the `eyes` chassis module. Each is a separate scene record in `luv_scenes` and a separate component in the registry.

### Scene 1: Prompt Playground

**Purpose**: Build, test, and refine generation prompts using chassis module parameters. The workbench for prompt development before scaling to bulk generation.

| Field | Value |
|-------|-------|
| slug | `prompt-playground` |
| category | `instrument` |
| surface | `dom` |
| required_modules | `['eyes']` (parameterized — works with any module) |
| status | `prototype` |
| component | `prompt-playground` |

**Input**: Module parameters (auto-composed into initial prompt)
**Output**: Saved prompt templates, small number of test generations

**Features:**
1. **Template editor** — Module parameters displayed as insertable variables. Starting prompt auto-generated from parameter values. User edits freely.
2. **Single-provider generation** — Pick a provider (HF/Replicate/Google), configure model + settings, generate 1-4 images from the current prompt.
3. **Result display** — Generated images shown in a grid. Click to inspect full-size.
4. **Annotation** — Mark results as good/bad/notes per image. This is lightweight — not the full reinforcement flow.
5. **Prompt versioning** — Save named prompt versions. Compare results across versions. Load previous versions.

**Toolbar extensions:**
- Left: provider selector dropdown, model selector
- Center: generate button with count selector (1/2/3/4)
- Right: prompt version indicator, save button

**Agent surface (read-only):**
- Current prompt text
- Active provider/model
- Number of generations in this session
- Latest result annotations

### Scene 2: Image Generation

**Purpose**: Bulk generation of eye image candidates from a proven prompt template.

| Field | Value |
|-------|-------|
| slug | `image-generation` |
| category | `generative` |
| surface | `dom` |
| required_modules | `['eyes']` |
| status | `concept` |
| component | `image-generation` |

**Input**: Prompt template (from prompt playground), module parameters
**Output**: N candidate images saved to module media with `artifact_type: 'candidate'`

**Features:**
1. **Prompt template selector** — Load a saved prompt template from the playground
2. **Batch configuration** — Number of images, provider, model, variation parameters (seed, guidance, etc.)
3. **Batch execution** — Generate N images with progress tracking
4. **Result gallery** — Grid of all generated candidates with generation metadata
5. **Quick filter** — Immediate yes/no/maybe triage (not full reinforcement)

**Toolbar extensions:**
- Left: template selector, batch size
- Center: generate batch button, progress bar
- Right: candidate count, filter toggles

### Scene 3: Image Reinforcement

**Purpose**: Evaluate, rank, and select canonical eye images from candidates.

| Field | Value |
|-------|-------|
| slug | `image-reinforcement` |
| category | `diagnostic` |
| surface | `dom` |
| required_modules | `['eyes']` |
| status | `concept` |
| component | `image-reinforcement` |

**Input**: Candidate images (artifact_type: 'candidate' from image-generation)
**Output**: Canonical images (artifact_type: 'canonical'), ranking metadata

**Features:**
1. **Candidate gallery** — All candidates for the module, filterable by generation batch
2. **Side-by-side comparison** — 2-up or 3-up comparison view
3. **Ranking interface** — Drag-to-rank or numerical scoring
4. **Reference overlay** — Compare candidates against existing reference images
5. **Promote to canonical** — Mark top-ranked images as canonical
6. **Module link annotation** — Tag which parameter aspects each image captures well

**Relationship to existing `reinforcement-review`:**
The existing scene handles review sessions with classification (me/not_me/skip). Image reinforcement is a distinct flow focused on ranking and selection rather than identity classification. They share some UI patterns (grid, detail inspector) but serve different purposes. Consider extracting shared components.

**Toolbar extensions:**
- Left: view mode toggle (grid/comparison/detail)
- Center: current selection count, promote button
- Right: filter by batch, sort by rank

## Pipeline Flow

```
┌──────────────────┐
│ Prompt Playground │ ← START HERE
│ (instrument)     │
└────────┬─────────┘
         │ prompt templates
         ▼
┌──────────────────┐
│ Image Generation │
│ (generative)     │
└────────┬─────────┘
         │ candidate images
         ▼
┌──────────────────────┐
│ Image Reinforcement  │
│ (diagnostic)         │
└────────┬─────────────┘
         │ canonical images
         ▼
┌──────────────────┐
│ Video Generation │
│ (temporal)       │
└────────┬─────────┘
         │ candidate videos
         ▼
┌──────────────────────┐
│ Video Reinforcement  │
│ (diagnostic)         │
└────────┬─────────────┘
         │ canonical videos
         ▼
┌──────────────────┐
│ 3D Generation    │
│ (spatial)        │
└────────┬─────────┘
         │ candidate 3D models
         ▼
┌──────────────────────┐
│ 3D Reinforcement     │
│ (diagnostic)         │
└────────┬─────────────┘
         │ canonical 3D model
         ▼
┌──────────────────┐
│ Rigging          │
│ (spatial)        │
└──────────────────┘
         │ rigged model
         ▼ DONE
```

Navigation is **free** — user can enter any scene at any time. The pipeline is a suggested flow, not an enforced one. Scenes degrade gracefully when upstream artifacts don't exist (show empty state with guidance on what to do first).

## Shared Infrastructure

### Provider Abstraction (`lib/luv/stage/providers/`)

Thin adapter layer for generation APIs:

```ts
interface GenerationProvider {
  id: string;
  name: string;
  models: GenerationModel[];
  generate(request: GenerationRequest): Promise<GenerationResult>;
  checkStatus?(jobId: string): Promise<JobStatus>;  // for async providers
}

interface GenerationRequest {
  model: string;
  prompt: string;
  negativePrompt?: string;
  count: number;
  width: number;
  height: number;
  parameters: Record<string, unknown>;  // model-specific
}

interface GenerationResult {
  images: GeneratedImage[];
  metadata: {
    provider: string;
    model: string;
    prompt: string;
    parameters: Record<string, unknown>;
    duration_ms: number;
    cost_estimate?: number;
  };
}
```

Provider implementations:
- `lib/luv/stage/providers/replicate.ts`
- `lib/luv/stage/providers/huggingface.ts`
- `lib/luv/stage/providers/google.ts`

### Prompt Composition (`lib/luv/template-engine.ts`)

Already exists. The prompt playground scene uses this to:
1. Build `TemplateContext` from chassis module parameters
2. Render `{{variable}}` templates with parameter values
3. Provide insertable variable list to the template editor UI

### Shared Scene UI Components (`app/(private)/tools/luv/stage/components/shared/`)

Reusable across scenes:

- `ImageGrid` — responsive grid with selection, zoom, annotation overlay
- `ComparisonView` — 2-up / 3-up side-by-side with synced zoom/pan
- `ArtifactPicker` — select artifacts from module media filtered by type
- `ProviderSelector` — dropdown for provider + model selection
- `GenerationProgress` — progress bar / status for async generation jobs
- `AnnotationOverlay` — draw/label/note on top of images or video frames

## Open Questions

1. **Prompt template storage**: Where do saved prompt templates live? Options: JSONB on `luv_scenes`, new `luv_prompt_templates` table, or within module parameters. Leaning toward a new table to support versioning and cross-scene reuse.

2. **Generation API route**: Should generation go through a single `/api/luv/generate` route that dispatches to providers, or separate routes per provider? Single route is simpler; separate routes allow provider-specific streaming patterns.

3. **Cost tracking**: Do we want to track generation costs? Provider APIs often expose token/credit usage. Could store in `generation_metadata`.

4. **Async generation**: Some providers (Replicate, HF) return job IDs and require polling. Need a polling/webhook pattern. Options: client-side polling, server-side polling with SSE, or Supabase Realtime on a jobs table.

5. **3D file storage**: GLB/GLTF files can be large. Storage bucket strategy — same `luv-images` bucket or a new `luv-models` bucket?

6. **Cross-module scenes**: The prompt playground works with any module (not just eyes). Should scenes be explicitly per-module or generic with module as a parameter? Current design: scenes declare `required_modules` but the component code is module-agnostic.
